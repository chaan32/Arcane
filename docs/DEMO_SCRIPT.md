# Arcane Demo Script

작성일: 2026-06-04

## 목표

시연은 "검색 서비스"가 아니라 "외부 API 기반 분석 플랫폼을 안정화하고 확장한 프로젝트"로 보여주는 것이 좋다.

권장 시연 시간:

- 짧은 버전: 5분
- 면접 버전: 10분
- 발표 버전: 15분

## 1. 오프닝

말할 내용:

> Arcane은 League of Legends 전적과 챔피언 분석을 제공하는 풀스택 프로젝트입니다. Riot API 기반 전적 검색, DB 캐싱, Redis 랭킹, OAuth 로그인, 공략 게시판, 실시간 채팅, AI 점수 모델 실험까지 구현했습니다.

강조:

- 외부 API 기반 서비스
- rate limit 대응
- 동시성/중복 저장 문제 해결
- 실시간 채팅
- AI는 실험 기능

## 2. 홈 검색

화면:

- 홈
- 검색창
- 최근 검색/즐겨찾기 드롭다운

시연:

1. 소환사명 입력
2. 최근 검색 드롭다운 확인
3. 즐겨찾기 토글 설명
4. `gameName#tagLine` 검색

말할 내용:

> 검색은 `gameName#tagLine`을 기준으로 처리하고, 최근 검색과 즐겨찾기는 프론트 localStorage에 저장합니다. 서버 DB까지 쓰지 않아도 되는 개인화 기능은 프론트 캐시로 분리했습니다.

## 3. 소환사 상세 페이지

화면:

- 프로필
- 랭크
- 전적 갱신 버튼
- 최근 갱신 시간
- 포지션 분포
- 전체 게임 요약
- 최근 많이 플레이한 챔피언
- 전적 카드

시연:

1. 프로필/랭크 확인
2. `최근 갱신` 배지 설명
3. 전적 갱신 버튼 설명
4. 전적 카드 확장
5. 소환사명 복사 버튼

말할 내용:

> 일반 검색은 `refresh=false`로 동작해서 DB에 있는 데이터는 Riot API를 다시 호출하지 않습니다. 사용자가 전적 갱신 버튼을 눌렀을 때만 `refresh=true`를 붙여서 Riot API를 강제로 호출합니다. 이를 통해 API 호출 수와 429 가능성을 줄였습니다.

백엔드 연결 설명:

> `summoner.puuid`와 `match_info.match_id`는 unique key라 동시에 저장하면 충돌이 날 수 있습니다. 그래서 중복 가능성이 있는 저장은 `INSERT IGNORE`로 처리하고, 저장 후 항상 DB에서 다시 조회해 후속 participant 저장에 사용했습니다.

## 4. 챔피언 분석

화면:

- 챔피언 분석 목록
- 전체/라인별 탭
- 챔피언 아이콘과 티어 표시
- 챔피언 상세

시연:

1. 전체 보기
2. 라인별 보기
3. 챔피언 상세 이동
4. 빌드/룬/패치 정보 탭

말할 내용:

> 챔피언 분석은 라인별 통계를 보여주고, 이미지 파일명은 Riot Data Dragon의 champion id/name 매핑을 사용합니다. 한글 챔피언명을 이미지 파일명으로 쓰면 깨지기 때문에 Data Dragon 기준 영문 id를 사용했습니다.

## 5. 패치노트

화면:

- 전체 패치노트
- 챔피언별 패치노트
- 공식 링크

시연:

1. 전체 패치노트 목록
2. 챔피언 검색
3. 공식 링크 클릭 가능함 설명

말할 내용:

> 공식 패치노트 페이지는 서버에서 매번 실시간 크롤링하면 connection reset이 발생할 수 있습니다. 그래서 백엔드에서 User-Agent를 명확히 넣고 retry, 캐시, fallback 목록을 둔 구조로 만들었습니다.

## 6. OAuth 로그인과 온보딩

화면:

- 로그인 버튼
- Google/Naver 선택
- OAuth callback
- 온보딩
- 내 정보

시연:

1. 로그인 버튼 설명
2. 최초 로그인 시 닉네임 온보딩 설명
3. `/me` 페이지에서 닉네임 변경

말할 내용:

## 8. 실시간 채팅

화면:

- 공략 상세의 메시지 보내기
- Header 채팅 dock
- 메시지 전송
- 읽음 표시
- unread badge
- 차단/삭제 버튼

시연:

1. 공략 작성자에게 메시지 보내기
2. 채팅 dock 열기
3. 메시지 전송
4. unread badge 설명
5. 차단/삭제 기능 설명

말할 내용:

> 채팅은 REST polling이 아니라 STOMP WebSocket으로 구현했습니다. 클라이언트는 `/pub/chat/rooms/{roomId}/messages`로 보내고, `/sub/chat/rooms/{roomId}`를 구독합니다. 서버에서는 Principal로 현재 사용자를 확인해서 해당 방 참가자인지 검증합니다.

주의:

- 현재 Redis Pub/Sub이나 Kafka 기반 채팅은 아니다.
- Spring simple broker 기반 STOMP라고 정확히 말한다.

## 9. AI 점수 실험

화면:

- 가능하면 AI 문서 또는 metrics 화면
- 전적 카드의 점수 영역

시연:

1. 현재 점수는 백엔드에서 랜덤 점수로 들어간다고 솔직히 설명
2. AI 서버에는 `/predict` 모델이 준비되어 있다고 설명
3. metrics를 설명

말할 내용:

> AI 점수는 현재 서비스 완성 기능이라기보다 실험 단계입니다. DB의 match participant 지표를 feature로 만들고 회귀 모델을 학습했습니다. 현재 clean rows는 약 1.5만 개이고, 목표 4만 row에는 부족합니다. 실제 서비스화 전에는 외부 label 리스크를 피하기 위해 Riot 공식 API 기반 자체 점수 공식으로 전환하는 것이 맞다고 판단했습니다.

## 10. 마무리

말할 내용:

> 이 프로젝트에서 가장 많이 신경 쓴 부분은 단순 기능 구현보다 외부 API 기반 서비스에서 생기는 실제 문제였습니다. Rate limit, 중복 저장, refresh 정책, WebSocket 인증, OAuth 계정 연결 같은 문제를 직접 겪고 구조를 개선했습니다.

## 예상 질문과 답변

질문: 왜 `refresh`를 나눴나요?

답변:

> 일반 조회와 강제 갱신의 목적이 다르기 때문입니다. 일반 조회는 빠르게 DB에서 보여주고, 사용자가 갱신을 원할 때만 Riot API를 호출하게 해서 rate limit을 줄였습니다.

질문: `INSERT IGNORE`를 왜 썼나요?

답변:

> 같은 match나 summoner가 동시에 저장될 수 있어서 unique key 충돌이 정상적으로 발생 가능한 구조였습니다. 예외를 잡는 것보다 DB 레벨에서 중복 insert를 무시하고 재조회하는 방식이 더 안정적이라고 판단했습니다.

질문: 채팅에 Redis Pub/Sub을 썼나요?

답변:

> 현재는 Spring simple broker 기반 STOMP입니다. 단일 인스턴스에서는 충분하고, 다중 인스턴스로 확장할 때 Redis Pub/Sub이나 broker relay를 붙이는 방향이 적절합니다.

질문: AI 모델은 실제 서비스에 들어갔나요?

답변:

> FastAPI `/predict`와 모델 파일은 준비되어 있지만, 현재 백엔드 전적 점수는 아직 `/random` 기반입니다. 다음 단계는 match participant feature를 백엔드에서 구성해 `/predict`로 보내는 것입니다.

질문: DeepLOL 데이터는 써도 되나요?

답변:

> 공개 포트폴리오에서는 조심해야 합니다. 기술 실험으로는 구현했지만, 서비스화나 공개 설명에서는 Riot 공식 API 기반 자체 점수 공식으로 전환하는 방향이 더 안전합니다.
